Project 02

Co-living Task & Split Bill Manager

Designing a co-living management app to resolve roommate conflicts through automated chore scheduling and transparent bill splitting.

Business AnalystRole
3 WeeksTimeline
Figma, Jira, TrelloTools

1. The Problem

Living with roommates often leads to friction over two common enterprise-scale issues: resource allocation (chores) and financial reconciliation (split bills). Typically, a whiteboard or a messy Zalo group chat is used, which lacks accountability and traceability.

Without a transparent system, roommates end up arguing over "Who cleaned last?" and "Who paid for the shared groceries?"

2. Constraints & Trade-offs

The Constraint: Developing a full-stack mobile application with real-time push notifications required significant backend infrastructure (Firebase/AWS) which was overkill for a 3-person apartment.

The Trade-off: Instead of building a complex notification engine, I designed a frontend-only MVP that generates a weekly "Chore & Ledger Summary" as an image. This image is automatically shared to our existing Zalo group chat via a simple bot webhook.

Why it works: It leverages the existing habit of checking Zalo, rather than forcing users to download and check a new app daily.

3. Approach & BA Artifacts

A. Requirements Elicitation
I interviewed my roommates to gather requirements, identifying two distinct personas: The "Clean Freak" (wants strict schedules) and the "Forgetful Payer" (needs auto-calculated debts).

B. User Stories
I mapped out Epics and User Stories. Example: "As a user, I want to add a shared grocery receipt so that the cost is automatically split 3-ways and added to the monthly ledger."

C. UI/UX Prototyping
I designed wireframes in Figma focusing on a dual-panel layout: a Kanban board for chores, and a Ledger table for transparent bill tracking.

4. Core System Functions & UI Design

Beyond the interface, I defined strict functional requirements for the MVP:

  • Chore Kanban Board: Visual task management with To Do, In Progress, and Done states.
  • "Auto-Assign" Algorithm: Fairly distributes tasks based on historical completion and availability constraints.
  • Split Bill Ledger: Centralized expense input and auto-calculated inter-personal debts.
  • Decimal & State Handling: System automatically rounds infinite decimals (e.g., $10/3 = $3.33 + $0.01 to payer) and securely tracks "Paid/Unpaid" statuses.

The solution is a clean, single-page web app tailored for mobile views but scalable to desktop, utilizing Bento Grid 2.0 layout principles.

Real-world UI/UX Benchmark

A co-living app must manage complex, disparate data streams (chores, ledgers, schedules). To maximize scanability, I adopted the Bento Grid 2.0 layout, inspired by top-tier tech companies:

  • Ramp & Apple (iOS 26): Influenced the strict modular grid with exaggerated corner radii and uniform gutters. The bento blocks dynamically stack and scale based on data importance and screen size.
  • WhatsApp (by Koto Studio): Inspired the compartmentalization of features into distinct, flexible modules, making dense roommate data visually harmonious.

5. BA Deliverables Deep Dive

A. User Stories & Acceptance Criteria

I utilized standard Agile documentation formats to ensure development alignment for the "Auto-Assign" feature.

User Story:

"As a Roommate, I want the system to auto-assign weekly chores so that responsibilities are shared fairly without manual arguments."

Acceptance Criteria (BDD Format):

  • Given the chore "Clean Bathroom" has not been assigned this week,
  • When the user clicks the "Auto-Assign" button,
  • Then the system assigns the chore to the roommate with the lowest historical completion count,
  • And updates the Kanban board to reflect the new assignee immediately.

C. Requirements Traceability Matrix (RTM)

To ensure no requirement was missed during development and testing, I mapped the business needs directly to test scenarios.

Req ID Business Requirement User Story Test Case ID

D. Test Case Design (Split Bill Ledger)

To demonstrate Requirement-Based Testing, I designed edge-case scenarios for the bill splitting algorithm to prevent logic failures.

Scenario Test Steps Expected Result
Infinite Decimal Split Input $10.00 split among 3 people. System rounds to 2 decimals ($3.33) and assigns the remaining $0.01 to the payer.
Negative Value (Exception) User inputs "-50" as an expense amount. System blocks input, triggers validation error: "Amount must be greater than zero."
Leave While in Debt (Edge Case) User attempts to "Leave Group" while owing $15.00. System blocks action, prompts: "Please settle outstanding debts before leaving."

6. Business Impact & Results

By treating a co-living space like a micro-enterprise, we resolved internal conflicts through logic and transparency.

  • Eliminated 100% of disputes regarding unpaid shared expenses.
  • Improved chore completion rate by gamifying the "Done" column on the Kanban board.

Key Takeaway: Excellent Requirement Elicitation involves understanding the true bottleneck (user habits). A perfectly coded app fails if users don't want to open it; integrating with their existing Zalo habit was the real win.
